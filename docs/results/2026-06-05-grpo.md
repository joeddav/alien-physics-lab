# 2026-06-05 — GRPO: procedural-diversity knob runs

First training runs using the three diversity knobs added 2026-06-04 (see that doc for the
design + the TRL tool-schema-frozen finding + the adversarial review). Goal: more task variety
without leaving the toy regime, and a cleaner handle on adaptive behavior.

## TL;DR
`--vary-tools` (G8, lr 2e-6) **broke the long-standing n=2 aggregation plateau** — experiments
rose 1.4 → **3.5** (%≥3: 8% → 92%) with healthy entropy and *no* collapse despite the lr bump.
The model also **learned to stop calling soft-disabled tools** (tools/failure_frequency 0.06 → 0).
Caveat: three things changed at once vs the G16 baseline (G8, lr 2e-6, vary-tools), so the escape
is not yet attributable to one. `--vary-precision` (isolated) is running next to test the clean
precision→aggregation hypothesis.

## `rl-vary-tools-g8` — details
- **Aggregation plateau broken.** Every prior run (lowlr-g16, varnoise-g16, shiftgeom) plateaued
  at ~2.0–2.2 experiments. This one: early 1.36 → mid 3.23 → late **3.54**; late distribution
  60.7% n=3, 18% n=4, with a tail to n=14. measurement_reward rose 0.02→0.20 (toward the 0.5 cap).
- **Entropy held under the lr bump.** 2e-6 (2× the 1e-6 baseline) did NOT collapse entropy (the
  failure mode at 1e-5): entropy rose 0.23→0.39 and settled ~0.29. So the safe-lr ceiling for this
  setup is comfortably above 2e-6 at G8.
- **Knob 2 adaptive signal confirmed.** tools/failure_frequency fell 0.06→0: the policy learned to
  read the briefing's "Available tools" list and stop calling soft-disabled experiments (it never
  saw a schema change — the tools are always advertised; it learned purely from the briefing text +
  the in-band error). Accuracy is comparable across tool regimes (drop-only 0.41, both 0.35,
  pendulum-only 0.37) — no degenerate single-tool worlds.
- **Confound.** vs the G16 baseline this changed G16→G8, lr 1e-6→2e-6, AND added vary-tools. The
  plateau escape is most plausibly the lr bump finally letting the (unchanged) geometric
  measurement reward drive aggregation, possibly helped by single-tool worlds forcing repeated use
  of one instrument. Not yet isolated — an ablation (vary-tools at lr 1e-6, or this lr with no knob)
  would separate them. Report: `out/grpo-rl-vary-tools-g8/aggregation_report.md`.

## `rl-vary-precision-g8` — knob doesn't elicit adaptation; the hint is moot (killed at step 132)
Ran knob 1 in isolation, then investigated a concern that the stated precision "gives too much
of a hint." Two findings, one expected and one more important:

1. **The hint is real (root cause = the noise-coupling).** A leak-quantification workflow confirmed:
   `tol = noise·mult` (mult~logU[0.8,3.0]) means the stated "within X%" is correlated with the hidden
   noise. Under *constant* noise (this run) it leaks 0 bits about noise but lets the model precompute
   `k=4/mult²` and skip closed-loop scatter assessment (~2.6 bits of "how much to aggregate" handed
   over). Under *varying* noise it's a partial readout of the hidden noise (3.75× window ≈ 1 bit / 34%
   of the noise-scale uncertainty). Being *told the precision* is a legitimate task input; the **leak
   is specifically the coupling** — fix = draw tolerance from an ABSOLUTE range independent of noise.
2. **But the knob doesn't work at all — and the hint is therefore moot.** Empirically (late half,
   264 rollouts): **Spearman ρ(tolerance, n_exp) = 0.05** and n_exp is dead flat across tolerance
   buckets — tight(<8%) 2.00, mid 1.99, loose(>13%) 2.02. The model does ~2 experiments **regardless
   of the stated tolerance**. Root cause (workflow critiques converged here): **experiments are free**
   (budget removed) and the measurement reward only credits *more*, so there is no incentive to do
   *fewer* on loose worlds or *more* on tight ones → the reward-optimal aggregation is a fixed amount,
   independent of tolerance. Decoupling the tolerance would kill the leak but NOT make the knob adapt.
   **Conditionable aggregation requires a per-experiment COST** (so over-measuring is penalized and the
   agent stops at the tolerance-justified k) — which re-introduces a soft version of the removed budget.
   Decision pending with the user. (Also note: this run sat at n_exp~2, i.e. it did NOT replicate
   vary-tools' jump to 3.5 — consistent with the plateau-break being the lr/G8, not the knob; the
   baseline ablation tests that directly.)

## `rl-baseline-g8-lr2e6` — the plateau-break was the lr, NOT the knob (confound resolved)
No-knob ablation, identical config to rl-vary-tools-g8 minus `--vary-tools`. Result: n_exp
1.93→2.76→**3.46** (%≥3 7%→67%), final tools/call_frequency **3.69**, physics→0.44, entropy 0.32 —
**statistically indistinguishable from vary-tools** (3.54 / 3.69 / 92% / ~0.53). So the long-standing
n=2 plateau was an **optimization/lr issue**, not a task-diversity one: lr-1e-6 runs (lowlr-g16,
varnoise) sat at ~2.1; **lr 2e-6 + G8 reaches ~3.5 regardless of any knob.** `--vary-tools` adds genuine
*input* diversity (and the model learns to avoid disabled tools) but does NOT drive aggregation depth.
Note it still settles ~3.5, well short of the idealized reward-optimum (~12) — and since truncation
is NOT binding (a measurement costs ~50 tok; completions ~1.5k/4096; length ⊥ n_exp, R²=0.01), the ~3.5
ceiling is a learned behavioral choice, not a budget limit. Pushing past it is a reward/exploration
question, not infrastructure.

## `rl-vary-tools-g8-1k` — learns a LOT, then DIVERGES (~step 380); fix = KL anchor
The 1000-step run answered "does it eventually learn?" emphatically, then revealed an instability:
- **It learns far more with steps.** By ~step 300 aggregation reached **n≈9 experiments** (vs 3.5 at
  200 steps) and reward ~1.2 — so the ~3.5 ceiling was just "not enough steps," not a hard limit.
  Combined with the ablation, the lr-2e-6 dynamic keeps pushing aggregation up given time.
- **Then it diverged (~step 380).** Entropy ran AWAY UPWARD: 0.5 → 0.8 → 1.6 (380) → 5.4 (442) → **8.7**
  (458); physics 0.62 → **0.0**, tools/call 9 → **0.5**, clipped 0 → **0.75** (policy emitting
  high-entropy garbage that truncates). Classic policy-gradient divergence: **`beta=0` (no KL anchor)
  + sustained lr 2e-6 → unbounded cumulative drift over a long run.** The default linear lr-decay was
  active and did NOT prevent it (still ~1.2e-6 effective at step 380). The 200-step runs never hit this
  only because they ended before enough drift accumulated.
- **Watchdog lesson:** mine keyed on entropy *< 0.05* (collapse); this was an explosion *upward*. Fixed
  `watch_run.py` to alarm on entropy **> 1.5** (divergence) as well.
- **Fix → `rl-vary-tools-g8-kl` (running):** add a KL penalty **beta=0.02** to anchor the policy to the
  base (directly opposes the entropy runaway). KL loads a reference model, so made room with per_device
  2 / grad_accum 4 (= G8) + vLLM fraction 0.18 — fits at 74 GiB, no OOM, kl≈4e-4 early. Open question:
  does KL hold the gains (n~9, reward ~1.2) without diverging, and without over-pulling aggregation back
  toward the base's n~1-2?
