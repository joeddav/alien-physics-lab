# 2026-06-04 — GRPO: measurement-reward design + the aggregation/exploration problem

Continues from `2026-06-03-grpo.md` (which fixed the frozen-policy weight-sync bug). Today's
thread: get the model to actually **aggregate multiple noisy measurements**, and understand
why it won't.

## TL;DR
After fixing real RL, the model reliably learns to do **~2 experiments but plateaus there**.
We worked through reward design (flat → geometric → shifted), hit two instructive failures
(a reward-shape backfire and an entropy collapse), and reframed the blocker as **exploration /
premature convergence**: GRPO can only reinforce action-counts it *samples*, and once the
policy sharpens around n=2 it never samples n>2, so the beyond-2 reward is never experienced.
Current run tests the fix (lower lr + more rollouts).

## Key lessons
1. **Reward shape matters more than magnitude.** The geometric reward *rewarding the 1st
   measurement* (0.15 at n=1) shrank the lazy-vs-diligent advantage gap → the policy drifted
   *down* to ~2 and entropy fell. Fix: reward only the **2nd+** measurement
   `reward(n)=cap·(1−decay^max(0,n−1))` — restores the must-aggregate cliff.
2. **lr is the dangerous knob; change one thing at a time.** Bumping lr 3×→1e-5 (to
   "compensate" for `scale_rewards=none`), stacked with 2 other changes and no KL/entropy
   guard, caused a full **entropy collapse** in ~20 steps. When trying `scale_rewards=none`,
   keep lr modest.
3. **`scale_rewards` is scale-invariant *within* a group** — it does NOT change the within-group
   signal-to-noise, so it can't, by itself, make the model do >2 experiments. It's a
   cross-group / gradient-variance lever (Dr-GRPO bias; stops blowing up pure-luck groups to
   unit variance), relevant to the broad luck-vs-skill problem, not the n=2 plateau.
4. **The n=2 plateau is an EXPLORATION problem.** GRPO can only reinforce action-counts it
   samples; if the policy rarely samples n>2 and updates sharpen it around 2, the beyond-2
   reward is never experienced → no gradient toward it. Magnitude is moot until it's sampled.
   → fix by preserving/adding exploration: **lower lr + more rollouts (larger G)**, and/or
   KL / DAPO clip-higher / higher temperature.

## Variance research (RLVR luck-vs-skill) — summary
Surveyed how RLVR counters reward variance from *environment/observation* stochasticity (a
lucky-measurement rollout gets high advantage without aggregating). Finding: almost all
GRPO-family variance work (Dr-GRPO, DAPO, RLOO, REINFORCE++, group-size) targets *baseline/
normalization* variance, not per-rollout observation luck. The families that actually attack
luck-vs-skill: (a) **resample-and-average the reward over K noise draws** (Monte-Carlo
denoising), (b) **reward design that credits aggregation / uncertainty-reduction** (what the
measurement reward attempts), (c) **true synchronized common-random-numbers** (key noise on
(tool,args,repetition), not call-order — our seed-sharing is true CRN only for byte-identical
traces). No single fix solves it; they stack. Caveat: "spurious rewards" — GRPO clipping can
produce gradients even from random reward, so validate with *behavioral* metrics (experiments/
rollout), not reward alone.

## Run log
- **`rl-varnoise-g16` OOM → relaunch** (memory boundary nailed down): the first diverse-task
  launch died on the backward pass (`tried to allocate 11.59 GiB, 6.24 GiB free`). Diffing the
  `training_args.bin` against the proven `rl-lowlr-g16-noise08` showed the **only** difference was
  `max_completion_length`: **5120 (OOM) vs 4096 (works)** — same per_device=4, ga=4, G16, lr 1e-6,
  vllm 0.22. So full-FT 1.7B at G16 sits right at the memory edge; a 25% longer completion tips it
  over (backward activations scale with batch×seqlen). Relaunched at 4096 (+ `PYTORCH_CUDA_ALLOC_CONF=
  expandable_segments:True` for fragmentation headroom): trains cleanly at ~94.5/97.9 GiB, ~103 s/step
  (200 steps ≈ 5.7 h). Lesson: **4096 is the completion ceiling for full-FT 1.7B @ G16**; to go longer
  either drop per_device to 2 (ga 8) or use LoRA. Early metrics: tools 2.8, physics 0.30, entropy 0.24,
  **but clipped_ratio 0.56** — the higher-noise worlds make the 1.7B deliberate more and hit the 4096 cap
  often (truncated → no boxed answer → physics 0), capping achievable reward. Watching whether it falls
  as behavior consolidates; if it stays high, the lever is per_device 2 + longer completion, not more memory.
- **Task too toy → varying noise** (key reframe): the task has effectively **one prompt** —
  ~8192 distinct *hidden-gravity* worlds, but the prompt/briefing is byte-identical for every
  one (and the world params were removed), so the model sees the same input every episode and
  there's a single optimal procedure. That narrowness underlies the n=2 plateau + exploration
  fragility. First fix (cheapest, most direct): **per-world varying hidden noise** (`--noise-min/
  --noise-max`, log-uniform [0.02,0.15]) so the optimal #measurements varies and the model must
  infer noisiness and adaptively aggregate. `rl-varnoise-g16` launched. (Bigger future axes:
  multiple hidden parameters / tool families; varied prompts/targets.)
- **`rl-lora4b-noise08`** (LoRA r32 Qwen3-4B, lr 1e-5, G8, 100 steps): learns (physics
  0.08→0.24, entropy 0.19→0.24, grad~0.1) but **did not beat the 1.7B (0.47)** — confounded by
  gentle LoRA lr × only 100 steps AND **severe truncation (clip 0.76→0.56): the 4B is far more
  verbose than the 1.7B and hits even the 6144 cap most of the time**, capping reward.
  Inconclusive on "is 4B better"; its verbosity is a real cost. (LoRA path itself works fine.)
- **LoRA + Qwen3-4B** (new axis): added `--lora` (peft LoraConfig, adapter merged→load_weights
  into vLLM each step) + `--vllm-max-model-length`. First smoke failed two ways, both fixed:
  (1) **disk quota** — 8 full-FT 1.7B runs each saved a ~6.5 GB fp32 `model.safetensors` (~52 GB);
  deleted all but the best (`lowlr-g16`) and the save crashed on EDQUOT. (2) The 4B is **much more
  verbose** than the 1.7B — it burned the 4096 completion budget on thinking + experiments and got
  truncated before emitting `\boxed` → all rewards 0 → zero advantage → grad 0 (NOT a LoRA bug).
  Fix: bigger completion (6144) + ctx (8192). `lora4b-smoke2` then **verified the LoRA path**
  (rewards climbed 0.025→1.8, grad>0, frac_zero_std→0, 4B boxes). Real run `rl-lora4b-noise08`

  measurements we likely also need a steeper beyond-2 reward and/or higher noise.
