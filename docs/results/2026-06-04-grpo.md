# 2026-06-04 — GRPO: measurement-reward design + the aggregation/exploration problem

Continues from `2026-06-03-grpo.md` (which fixed the frozen-policy weight-sync bug). Today's
thread: get the model to actually **aggregate multiple noisy measurements**, and understand
why it won't.

## TL;DR
After fixing real RL, the model reliably learns to do **~2 experiments but plateaus there**.
We worked through reward design (flat → geometric → shifted), hit two instructive failures
(a reward-shape backfire and an entropy collapse), and reframed the blocker as **exploration /
premature convergence**: GRPO can only reinforce action-counts it *samples*, and once the
policy sharpens around n=2 it never samples n>2, so the beyond-2 reward is never experienced.
Current run tests the fix (lower lr + more rollouts).

## Key lessons
1. **Reward shape matters more than magnitude.** The geometric reward *rewarding the 1st
   measurement* (0.15 at n=1) shrank the lazy-vs-diligent advantage gap → the policy drifted
   *down* to ~2 and entropy fell. Fix: reward only the **2nd+** measurement
   `reward(n)=cap·(1−decay^max(0,n−1))` — restores the must-aggregate cliff.
2. **lr is the dangerous knob; change one thing at a time.** Bumping lr 3×→1e-5 (to
   "compensate" for `scale_rewards=none`), stacked with 2 other changes and no KL/entropy
   guard, caused a full **entropy collapse** in ~20 steps. When trying `scale_rewards=none`,
   keep lr modest.
3. **`scale_rewards` is scale-invariant *within* a group** — it does NOT change the within-group
   signal-to-noise, so it can't, by itself, make the model do >2 experiments. It's a
   cross-group / gradient-variance lever (Dr-GRPO bias; stops blowing up pure-luck groups to
   unit variance), relevant to the broad luck-vs-skill problem, not the n=2 plateau.
4. **The n=2 plateau is an EXPLORATION problem.** GRPO can only reinforce action-counts it
   samples; if the policy rarely samples n>2 and updates sharpen it around 2, the beyond-2
   reward is never experienced → no gradient toward it. Magnitude is moot until it's sampled.
   → fix by preserving/adding exploration: **lower lr + more rollouts (larger G)**, and/or
   KL / DAPO clip-higher / higher temperature.

## Variance research (RLVR luck-vs-skill) — summary
Surveyed how RLVR counters reward variance from *environment/observation* stochasticity (a
lucky-measurement rollout gets high advantage without aggregating). Finding: almost all
GRPO-family variance work (Dr-GRPO, DAPO, RLOO, REINFORCE++, group-size) targets *baseline/
normalization* variance, not per-rollout observation luck. The families that actually attack
luck-vs-skill: (a) **resample-and-average the reward over K noise draws** (Monte-Carlo
denoising), (b) **reward design that credits aggregation / uncertainty-reduction** (what the
measurement reward attempts), (c) **true synchronized common-random-numbers** (key noise on
(tool,args,repetition), not call-order — our seed-sharing is true CRN only for byte-identical
traces). No single fix solves it; they stack. Caveat: "spurious rewards" — GRPO clipping can
produce gradients even from random reward, so validate with *behavioral* metrics (experiments/
rollout), not reward alone.

## Procedural-diversity knobs (the "effectively one prompt" fix)

**Problem.** All task diversity reaches the model through *tool-return values* only: the
briefing is byte-identical every episode (constant SYSTEM/USER prompts; `instructions()` had
no per-world params), and just the hidden latent (g, noise) varies. Two worlds look identical
until the agent acts. Three layers where diversity *could* live — (1) latent/tool-output [all
of it today], (2) prompt/framing [unused], (3) task structure [unused] — so we added knobs in
layers 2–3. Built after a design+investigation workflow (TRL tool-mechanism probe + adversarial
interaction review). All default OFF (byte-identical to before, locked by `tests/test_diversity.py`),
all CRN-safe (each per-world value is a deterministic `f(seed)` drawn in `make_dataset` with a
distinct decorrelated RNG, so every rollout in a GRPO group shares it — verified group-constant
in the live parquet).

- **Knob 1 `--vary-precision`** (task-structure + input). Per-world required precision is **stated**
  in the briefing ("within X%") and used by `score_answer` (with a proportional zero-band,
  `REWARD_ZERO_RATIO=4`). Tolerance is drawn as `noise · U[0.8, 3.0]`, **not** an absolute value:
  since g comes from a time measurement, `rel_err(g) ≈ 2·noise` and averaging k trials shrinks it
  `~1/√k`, so the required aggregation `k ≈ 4/mult²` is **decoupled from noise** and every world is
  winnable within budget. (The design review caught that an *absolute* range makes the tight half
  structurally unwinnable at higher noise → all-zero groups + a success-bonus lottery; the
  noise-relative draw fixes it.) This is the *clean, measurable* version of the aggregation thread:
  the needed effort is visible in the prompt (conditionable), and pairing with `--noise-min/max`
  forces online noise-estimation (noise hidden, tolerance stated).
- **Knob 2 `--vary-tools`** (task-structure + input). Per-world subset {both | drop_ball | pendulum}.
  **Mechanism = soft-disable**: TRL freezes the advertised tool schema once from env instance 0
  (`grpo_trainer.py:523`) and `reset()` cannot mutate it, so per-world schema filtering is impossible
  without a fragile monkeypatch. Instead the briefing lists only the allowed experiments and a
  disabled call returns an in-band `{"error": ...}` **without touching the lab** (no noise-stream
  advance, no `tool_calls` increment — tested). Smoke showed `tools/failure_frequency ~0.25` from
  disabled-tool attempts, handled cleanly. calculator always available.
- **Knob 3 `--vary-prompt`** (input/robustness). Per-world scenario-framing intro (4 templates;
  index 0 = today). **Only the intro paragraph swaps**; the tool list, precision clause, and boxed
  directive are byte-identical across templates (no hint-leak surface, parsing unaffected). One
  reviewer argued to drop it as orthogonal/maintenance-heavy; kept per the user's "do all three"
  but scoped to intro-only to defuse the byte-fragility/hint-leak risk.
