# 2026-06-03 — GRPO setup + overnight sweep

## TL;DR
Stood up multi-turn GRPO (TRL v1) on the alien physics lab, on a single RTX PRO 6000
Blackwell (96 GB, SM120). The full pipeline runs end-to-end and **learns** (gradients
flow). Ran an overnight sweep over Qwen3-1.7B. Setup details: `docs/grpo_training.md`.

**Headline finding (REVISED after analysis — see "Where do the gains come from?"):**
GRPO produces a **small, noisy** within-run reward improvement, not the clean win the
first draft of this doc claimed. Authoritative full-batch trend (soak-lr3e6):
`physics_reward/mean` ~0.47→0.51, slope +0.13/100 steps, r=0.20, non-monotonic. The
earlier "~0.40→~0.48" cross-run number was **confounded** by a `max_completion_length`
change (4096→5120) and by the 8-rollout completion sample overstating the full-batch
trend. The gain is **not** from thinking shorter, **not** from more valid completions
(submission already ~96%), and **not** from measurement aggregation (experiments/rollout
flat ~1.2); the residual is marginally better single-shot accuracy. Reward-hacking
(submitting the ~9.8 Earth prior) persists at both ends. Best checkpoint:
`out/grpo-soak-lr3e6/`.

## What was set up
- Isolated training venv `/workspace/trl-grpo-venv`; pinned stack (torch 2.11+cu129,
  vLLM 0.22.1 nightly, transformers 5.9.0, TRL 1.5.1 no-`[vllm]`-extra).
- `grpo_env.py` (TRL `environment_factory` wrapper: tools + `submit_answer` + rewards),
  `grpo_data.py` (one hidden world per row; within-group same-world determinism),
  `scripts/train_grpo.py` (presets + sweep overrides + SM120 shims).
- Model: **Qwen3-1.7B**, thinking ON, full bf16, colocated vLLM rollouts.

## Validation (smoke runs)
The smoke is `--preset smoke` (Qwen3-1.7B, G=4, 8 steps). Three iterations, each fixing
a real issue — the loop ran end-to-end every time; these were tuning/correctness fixes:

| Issue | Symptom | Fix |
|---|---|---|
| Standalone vLLM on SM120 | — | cu13 `libcudart` shim + `enforce_eager`; generates correctly |
| Colocate OOM | `CUDA OOM`, 86/95 GB used | `vllm_enable_sleep_mode=True`, vLLM frac 0.45→0.30, grad-ckpt, expandable_segments |
| **Zero gradient** | `grad_norm≈1e-10`, `loss≈0`, `IS_ratio≈1e-26` | `vllm_importance_sampling_mode="token_truncate"` (was `sequence_mask`) |

After the fixes: 8/8 steps, `grad_norm≈1.9–2.95`, `reward` swinging 0.05→1.20 with
`reward_std>0` (groups have learning signal), `frac_reward_zero_std=0` on most steps.
Model thinks → runs 2–10 tool calls → `submit_answer`; `physics_reward/mean` hit 1.099
on a group where most rollouts succeeded. **Conclusion: the loop is correct and learns.**

Notes for real runs: thinking traces are long → 25–75% of completions hit the 4096-token
smoke cap (no clean EOS). **Correction (see analysis below):** "clipped" (hit cap) ≠
"scored 0" — most clipped rollouts had already called `submit_answer` and scored; the true
non-submit rate is only ~4%. So truncation is NOT the reward bottleneck the early run-log
entries assumed. Real runs still use a larger `max_completion_length` (more room → modestly
better-reasoned answers), but that is an accuracy effect, not a finishing effect.

## Overnight sweep (Qwen3-1.7B, full bf16, thinking ON)
Sequential runs on the one GPU; each writes `out/grpo-<name>/log_history.json` + final
model, documented + committed atomically on completion. Plan is adaptive — later runs may
change based on earlier results. Base config: `--preset real --per-device-batch 8
--grad-accum 1 --num-generations 8` (8 rollouts/step), `max_completion_length=6144`.

Memory note: `per_device=8 × 6144` OOMs on the backward pass; the safe profile is
`per_device_train_batch_size=4, gradient_accumulation_steps=2` (group=8 via accumulation,
peak tracks the 4-seq micro-batch), `max_completion_length=4096`, `vllm_gpu_memory_utilization=0.22`.

## Run log
- **13:38–14:23 — `nothink-lr3e6`** (150 steps, lr 3e-6, thinking OFF): exit 0.
  **Decisive comparison:** non-thinking `physics_reward` ~0.04→0.09 vs thinking ~0.47–0.51
  (~10× worse). Completions ~193 tokens, `clipped_ratio = 0.000` (no truncation at all),
  validity ~0.097 (submits + experiments fine). So the thinking trace's *reasoning content*
  (formula recall + arithmetic) does essentially all the work; without it the model can't
  infer gravity. This **doubly refutes** the "gains are just thinking-shorter / dodging
  truncation" worry — the no-thinking model cannot truncate yet performs terribly. step_time
  ~15s (~8× faster). GRPO still nudges it 0.04→0.09 (relative doubling off the floor).
- **10:20–13:36 — `soak-lr3e6`** (90 steps, lr 3e-6, len 5120, thinking on): exit 0,
  **final model saved** (`out/grpo-soak-lr3e6/`). **Best so far.** Raising the completion
  budget 4096→5120 cut truncation ~42%→~26%. `physics_reward/mean` by 5ths
  0.42→0.51→0.42→0.57→0.50 (overall mean **0.48**, up from ~0.45 @run2 / ~0.40 @run1;
  peak **1.066** on a fully-solved group). entropy stable ~0.20–0.23; grad_norm ~1.66.

- **07:15–10:18 — `lr3e6`** (100 steps, lr 3e-6, G=8, thinking on): exit 0. **Clearer

  **Takeaway:** lr 3e-6 learns modestly; truncation is the bottleneck → run 3 raises

- **05:25–07:13 — `base-lr1e6`** (60 steps, lr 1e-6, G=8, thinking on): exit 0. Training

## Where do the gains come from? (analysis)
Investigated the source of the reward gain (per-completion parquets, early steps 2–20 vs
late 72–90 in the fixed-config `soak-lr3e6` run; full-batch `log_history` trends; reading
rollouts; adversarial adjudicator). The dated run-log "takeaways" above (truncation is the
bottleneck; clean 0.40→0.48 win) **do not survive**; this is the corrected picture.

Decomposition of the (small) gain:
- **Not thinking shorter / dodging truncation.** Within soak-lr3e6, mean completion length
  is flat (3823→3831), and the no-EOS clipped rate is flat (~0.24, slope ≈0). ("Clipped" ≠
  scored 0; true non-submit rate ~4%.) **Corroborated by run 4:** the non-thinking model
  has zero truncation (clipped=0) yet scores only ~0.07 vs ~0.5 with thinking — so the
  thinking *content* drives reward and the gain is not a truncation/length artifact.
- **Not more valid completions.** Submission rate already ~96–99% (ceiling) → decomposition
  attributes only ~6% of the gain to submission rate.
- **Not measurement aggregation** (the env's intended skill). Experiments/rollout flat-to-
  down (~1.2; frac ≥2 experiments fell 0.38→0.31); repeat-and-average ~0–2% throughout;
  reward vs #experiments r≈0.09 (n.s.). Genuine measure-twice-and-average shows up in only
  ~1–2 of 8 rollouts, equally rare early/late. The `tools/call_frequency` rise (+0.8) is
  calculator/submit churn, not more physics experiments.
- **Residual = marginally better single-shot accuracy** among submitters (right arithmetic /
  trusting the measurement vs the 9.8 prior). Shapley split of the parquet gain ≈ 94%
  accuracy / 6% finishing.

**Effect size is small and noisy.** Full-batch (authoritative) trend: physics_reward
0.47→0.51, slope +0.13/100 steps, r=0.20, non-monotonic (success-among-submitted tertiles
0.32→0.16→0.36 ≈ flat with a mid-run dip). The parquet 8-rollout bins (+0.21) overstate the
full-batch (+0.05) and don't even match the logged per-step metric (subsample / step-
alignment) — treat parquet magnitudes as directional only. One seed, 90 steps: a ~+0.05
drift is barely above noise. **Reward-hacking persists** (submitting ~9.8 despite contrary
measurement; zero-experiment submits) at ~4–5/16 rollouts at both ends.

**To get an honest answer next:** fixed held-out eval (same seeds, early vs late
checkpoint) to remove per-step task-difficulty noise; more steps + multiple seeds;
reconcile parquet-vs-log reward; consider penalizing the 9.8 prior. The non-thinking run
(#4) is an orthogonal check — if reward rises similarly without a thinking block, that
further rules out thinking-time effects.
