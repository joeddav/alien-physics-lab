# 2026-06-06 ‚Äî GRPO: KL-beta tuning for stable-but-still-learning long runs

Continues the long-run stability thread from `2026-06-05-grpo.md` (the 1000-step `--vary-tools`
run learned to n‚âà9 / reward ~1.2 by ~step 300 then DIVERGED ‚Äî entropy runaway ‚Äî because
`beta=0`). The fix was a KL anchor; this doc tunes its strength.

## TL;DR
`beta` is the only real entropy/stability knob in GRPO (KL anchor to base; no entropy term in
the loss). Swept it: **`beta=0` diverges** (entropy ‚Üí 8.7), **`beta=0.02` over-anchors** (stable
but plateaus ~0.63, aggregation *erodes* to ~3.5), and **`beta=0.01` is the sweet spot** ‚Äî full
1000 steps, **zero divergence** (max entropy 0.77), reward **0.90**, aggregation recovered to
**n‚âà5‚Äì6** (still climbing at the end) with a tail to 12. So a *light* anchor stabilizes without
strangling learning. Now probing `beta=0.005` to see if even less anchor reaches the no-KL's
n‚âà9 while staying stable, or finally tips back into divergence (bracketing the sweet spot).

## What we learned about KL/entropy in GRPO (TRL 1.5.1)
- **Entropy is NOT in the GRPO loss.** No entropy bonus, no entropy coefficient (verified in
  `grpo_config.py`); `train/entropy` is a logged diagnostic only. The one optional hook is
  `top_entropy_quantile` (<1.0 masks the loss to high-entropy tokens; default 1.0 = off).
- The loss is `‚àí(advantage √ó clipped_ratio) + Œ≤¬∑KL(œÄ‚ÄñœÄ_ref)`. So **the KL term (Œ≤) is the de-facto
  entropy regulator** ‚Äî indirectly, by anchoring the policy to the base (whose entropy is normal),
  preventing both collapse and runaway. With `Œ≤=0` (TRL default) nothing constrains entropy ‚Üí free
  drift ‚Üí the runaway we hit. Œ≤ is the only direct knob for it (besides lr/lr-decay shrinking updates).
- This reframes the beta sweep: we're tuning *how hard to anchor to base*. Too hard (0.02) ‚Üí drags
  aggregation back to base behavior; too soft (0) ‚Üí diverges. 0.01 tests the middle.

## Note: the reward oscillation is sampling noise, not instability
The train/reward curve swings ¬±0.2+ step-to-step. Decomposed it (KL run, 290 steps): per-step
reward std 0.226, of which ~0.12 is within-world (only **8 rollouts of ONE seed per optimizer
step** ‚Äî G8 with gen_batch=8=G ‚Üí 1 world/step) and ~0.19 is between-world (each step is a
different hidden world of different difficulty). Almost all of it flows through the *physics*
(accuracy) term (std 0.20); measurement/validity are smooth. It's **stationary** (amplitude
0.25‚Üí0.20 across halves) around a *rising* smoothed trend ‚Äî i.e. noise, not instability (real
instability = growing oscillation + diverging trend, like the entropy runaway). Lever to smooth it
AND lower gradient variance: **more worlds per optimizer step** (gen_batch = k√óG via grad_accum),
not lr/KL. Optional, modest; deferred.

## `rl-vary-tools-g8-kl01` (beta=0.01) ‚Äî the result
Full 1000 steps, exit 0. **Max entropy 0.766** over the whole run (never near the 1.5 alarm) ‚Äî no
divergence. Final (last 30): reward **0.896**, physics **0.495**, entropy 0.41, tools/call **6.0**,
kl 0.35. Aggregation trajectory **n_exp 1.2 ‚Üí 3.8 ‚Üí 5.4** (early‚Üímid‚Üílate), late distribution peaks
at n=4‚Äì5 (24%/24%) with a real tail (6:16%, 7:10%, 8:4%, 9:2%, 10‚Äì12:~7%), still rising at the end.
By tool subset: drop-only 3.5 / both 3.8 / pendulum-only 3.3 n_exp, accuracy ~0.37‚Äì0.41 ‚Äî comparable
across regimes. So a *light* KL anchor (0.01) buys stability AND keeps ~all of the learning the
no-KL run had before it blew up. This is the working long-run recipe.

## Reward decoupling: more measurements ‚âÝ better accuracy (why measurement‚Üë physics‚Üì)
Investigated a puzzle: over recent steps the measurement reward rose while physics fell, which is
backwards (more measurement should help accuracy). Traces from rl-vary-tools-g8-kl005 (recent 64
rollouts) show two things:
- **Partly a sampling artifact** (1 world/step, see fix below): physics is strongly gravity-dependent
  (low-g 0.69 ‚Üí very-high-g 0.28), and the recent window happened to draw mostly hard high-g worlds.
- **The real issue ‚Äî `corr(n_experiments, physics_reward) ‚âà ‚àí0.03` (zero).** More experiments are NOT
  improving accuracy. Traces: the high-n rollouts do height *sweeps* (drop from 1,2,4,‚Ä¶,100 m), not
  repeated reads averaged for variance reduction, and systematically **over-estimate** (+6‚Äì8% on the
  12‚Äì16 g band; two failures over-shot g‚âà20 ‚Üí 22.7, 24.5). So the measurement reward (count-based, gated
  only on producing *an* answer ‚Äî accuracy-blind) rises mechanically as aggregation increases, but
  physics doesn't follow because the bottleneck is the **aggregation/computation step**, not the count.
  A mild Goodhart: the model farms the measurement bonus via sweeps without converting them to accuracy.
- Implication: the measurement reward should probably condition on accuracy/uncertainty-reduction (or be
  dialed back now that the model over-aggregates to n‚âà8); and hard-world accuracy is gated by computation
  (the over-estimation), not by measurement count.

## One world per optimizer step ‚Üí four (the multi-world fix)
**Latent flaw found:** every run so far trained on ONE world per optimizer step ‚Äî `per_device(2) √ó
grad_accum(4) √ó num_procs(1) = 8 = num_generations`, so the whole generation batch was a single group =
one prompt. Each update was dominated by one world's idiosyncrasies ‚Üí high-variance gradient and the wild
train/reward oscillation (the ¬±0.2 step-to-step swing decomposed earlier is mostly *between-world*
variance, 1 world/step). Unless the goal is to overfit one world (which would be one world per *entire
run*), multiple worlds per step is the correct default.
**Fix (launch flag only, no code):** worlds/step = `per_device √ó grad_accum √ó num_procs / num_generations`
(TRL `RepeatSampler`: `generation_batch_size / num_generations` distinct prompts, `grpo_trainer.py:944`).
Bump `grad_accum 4 ‚Üí 16` (keep per_device 2, G8) ‚Üí generation batch 32 ‚Üí **4 distinct worlds √ó 8 gens per
step**. grad_accum is sequential so memory is unchanged (75.8 GiB, no OOM). Verified empirically: one
logged step's parquet now has 32 rollouts spanning 4 distinct world_gravity values (6.06/17.64/19.44/21.55).
`rl-mw4-kl005` running (beta 0.005, 250 steps); expect a much smoother reward curve and a lower-variance,
less idiosyncratic gradient. (Run *before* adding the new spin/diameter task, per the plan.)

**Result ‚Äî fix validated.** Reward step-to-step jitter HALVED (std 0.29‚Üí0.14; mean \|Œîstep\| 0.23‚Üí0.12 ‚Äî
the ‚âà1/‚àö4 expected from averaging 4 worlds), grad_norm dropped ~2√ó (the gradient is far less
idiosyncratic), and it was the most stable run yet (max entropy 0.357, no divergence). It reached
reward 0.76 / aggregation n‚âà4.7 at step 250 ‚Äî *below* the 1-world run's 0.88/n7.7, but only because it
had 250 optimizer updates vs 704 (each step is ~4√ó slower since it generates 4√ó the rollouts). So
multi-world is the correct setup with a cleaner per-update signal; the only cost is throughput.
**The payoff to cash in:** a 4√ó lower-variance gradient should tolerate a higher learning rate (we'd been
pinned at lr 2e-6 by *1-world* gradient noise). `rl-mw4-lr4e6` tests lr 4e-6 on the multi-world setup ‚Äî
if it learns faster per step while staying stable, the throughput penalty disappears. Adopt multi-world
as the standard batching going forward (and for the new task).

**Result ‚Äî recipe confirmed.** lr 4e-6 on multi-world reached reward **1.11** / physics **0.60** /
aggregation **n‚âà9.15** in just 200 steps ‚Äî the best of any run, ~3‚Äì4√ó faster than lr 2e-6, and STABLE
(entropy rose to ~0.6 mid-run then settled to 0.27; max 0.597, no divergence). So the cleaner gradient
genuinely tolerates the higher lr, more than erasing the 4√ó-slower-steps cost. **Standing recipe for all
future runs: 4 worlds/step (grad_accum 16, per_device 2, G8) + beta 0.005 + lr 4e-6.**
