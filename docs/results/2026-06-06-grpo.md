# 2026-06-06 — GRPO: KL-beta tuning for stable-but-still-learning long runs

Continues the long-run stability thread from `2026-06-05-grpo.md` (the 1000-step `--vary-tools`
run learned to n≈9 / reward ~1.2 by ~step 300 then DIVERGED — entropy runaway — because
`beta=0`). The fix was a KL anchor; this doc tunes its strength.

## TL;DR
`beta` is the only real entropy/stability knob in GRPO (KL anchor to base; no entropy term in
the loss). Swept it: **`beta=0` diverges** (entropy → 8.7), **`beta=0.02` over-anchors** (stable
but plateaus ~0.63, aggregation *erodes* to ~3.5), and **`beta=0.01` is the sweet spot** — full
1000 steps, **zero divergence** (max entropy 0.77), reward **0.90**, aggregation recovered to
**n≈5–6** (still climbing at the end) with a tail to 12. So a *light* anchor stabilizes without
strangling learning. Now probing `beta=0.005` to see if even less anchor reaches the no-KL's
n≈9 while staying stable, or finally tips back into divergence (bracketing the sweet spot).

## What we learned about KL/entropy in GRPO (TRL 1.5.1)
- **Entropy is NOT in the GRPO loss.** No entropy bonus, no entropy coefficient (verified in
  `grpo_config.py`); `train/entropy` is a logged diagnostic only. The one optional hook is
  `top_entropy_quantile` (<1.0 masks the loss to high-entropy tokens; default 1.0 = off).
- The loss is `−(advantage × clipped_ratio) + β·KL(π‖π_ref)`. So **the KL term (β) is the de-facto
  entropy regulator** — indirectly, by anchoring the policy to the base (whose entropy is normal),
  preventing both collapse and runaway. With `β=0` (TRL default) nothing constrains entropy → free
  drift → the runaway we hit. β is the only direct knob for it (besides lr/lr-decay shrinking updates).
- This reframes the beta sweep: we're tuning *how hard to anchor to base*. Too hard (0.02) → drags
  aggregation back to base behavior; too soft (0) → diverges. 0.01 tests the middle.

## Note: the reward oscillation is sampling noise, not instability
The train/reward curve swings ±0.2+ step-to-step. Decomposed it (KL run, 290 steps): per-step
reward std 0.226, of which ~0.12 is within-world (only **8 rollouts of ONE seed per optimizer
step** — G8 with gen_batch=8=G → 1 world/step) and ~0.19 is between-world (each step is a
different hidden world of different difficulty). Almost all of it flows through the *physics*
(accuracy) term (std 0.20); measurement/validity are smooth. It's **stationary** (amplitude
0.25→0.20 across halves) around a *rising* smoothed trend — i.e. noise, not instability (real
instability = growing oscillation + diverging trend, like the entropy runaway). Lever to smooth it
AND lower gradient variance: **more worlds per optimizer step** (gen_batch = k×G via grad_accum),
not lr/KL. Optional, modest; deferred.

## `rl-vary-tools-g8-kl01` (beta=0.01) — the result
Full 1000 steps, exit 0. **Max entropy 0.766** over the whole run (never near the 1.5 alarm) — no
divergence. Final (last 30): reward **0.896**, physics **0.495**, entropy 0.41, tools/call **6.0**,
kl 0.35. Aggregation trajectory **n_exp 1.2 → 3.8 → 5.4** (early→mid→late), late distribution peaks
at n=4–5 (24%/24%) with a real tail (6:16%, 7:10%, 8:4%, 9:2%, 10–12:~7%), still rising at the end.
By tool subset: drop-only 3.5 / both 3.8 / pendulum-only 3.3 n_exp, accuracy ~0.37–0.41 — comparable
across regimes. So a *light* KL anchor (0.01) buys stability AND keeps ~all of the learning the
no-KL run had before it blew up. This is the working long-run recipe.

## New-task feasibility: gpt-4o-mini benchmark (spin + diameter)
Before building GRPO training for the proposed spin/diameter tasks, validated they're well-posed
with a capable model. Added the instruments to env.py (benchmark scope only — no training plumbing):
`measure_horizon_dip` (diameter; dip=acos(R/(R+h)), g-independent), `foucault_pendulum` (precession
= Ω·sin φ, deg/hr), `star_tracker` (pole altitude = latitude ONLY, so spin must be *solved* not read),
calculator gains sin/cos/radians/etc, and a target-agnostic `score_value`. `scripts/bench_openai.py`
runs gpt-4o-mini via function-calling with target-aware briefings.

Result (10 episodes/target, noise 0.03, success = rel_err ≤ 3%):
| target | solve | median rel_err | mean tools |
|---|---|---|---|
| gravity (baseline) | 90% | 1.7% | 11.2 |
| diameter | 60% (+1 no-answer) | 1.0% | 4.8 |
| spin | 70% | 0.9% | 2.0 |

**Verdict: all three are well-posed and the right difficulty.** gpt-4o-mini's *reasoning* is correct
on every target (median errors ≤1.7%; spin's coupled Foucault+latitude→Ω→day chain works at 0.9% median).
The gap to 100% is the SAME on all: **under-averaging** — spin takes only 2 tool calls (one read each, no
repeats), diameter ~5, so unlucky single-read noise (3–6%) tips past 3%. The misses are noise, not
reasoning errors. That's exactly the aggregation skill GRPO instills, so these are well-suited RL targets:
a strong model is near-solving by reasoning, and the headroom is the behavior RL teaches. Notable design
wins confirmed: star_tracker-returns-latitude-only forces the spin combine (no transcription cheat); the
calculator trig + "α in radians" briefing avoided the ~3283× deg/rad blow-up; log-uniform ranges make
prior-mean guessing worthless. One diameter episode hit max_turns without boxing (looping) — a real but
rare (1/10) failure mode.

Implication: proceed to add the tasks for training. Diameter is the clean first target (1 tool, 1 formula,
g-independent); spin is viable as the richer coupled task (gpt-4o-mini clears the reasoning bar a 1.7B will
find harder — the right kind of hard for GRPO). Build = the staged plan in the feasibility workflow
(Stage 0 scorer generalization already partly done; add the GRPO/grpo_data/train_grpo plumbing next).

## Reward decoupling: more measurements ≠ better accuracy (why measurement↑ physics↓)
Investigated a puzzle: over recent steps the measurement reward rose while physics fell, which is
backwards (more measurement should help accuracy). Traces from rl-vary-tools-g8-kl005 (recent 64
rollouts) show two things:
- **Partly a sampling artifact** (1 world/step, see fix below): physics is strongly gravity-dependent
  (low-g 0.69 → very-high-g 0.28), and the recent window happened to draw mostly hard high-g worlds.
- **The real issue — `corr(n_experiments, physics_reward) ≈ −0.03` (zero).** More experiments are NOT
  improving accuracy. Traces: the high-n rollouts do height *sweeps* (drop from 1,2,4,…,100 m), not
  repeated reads averaged for variance reduction, and systematically **over-estimate** (+6–8% on the
  12–16 g band; two failures over-shot g≈20 → 22.7, 24.5). So the measurement reward (count-based, gated
  only on producing *an* answer — accuracy-blind) rises mechanically as aggregation increases, but
  physics doesn't follow because the bottleneck is the **aggregation/computation step**, not the count.
  A mild Goodhart: the model farms the measurement bonus via sweeps without converting them to accuracy.
- Implication: the measurement reward should probably condition on accuracy/uncertainty-reduction (or be
  dialed back now that the model over-aggregates to n≈8); and hard-world accuracy is gated by computation
  (the over-estimation), not by measurement count.

## One world per optimizer step → four (the multi-world fix)
**Latent flaw found:** every run so far trained on ONE world per optimizer step — `per_device(2) ×
grad_accum(4) × num_procs(1) = 8 = num_generations`, so the whole generation batch was a single group =
one prompt. Each update was dominated by one world's idiosyncrasies → high-variance gradient and the wild
train/reward oscillation (the ±0.2 step-to-step swing decomposed earlier is mostly *between-world*
variance, 1 world/step). Unless the goal is to overfit one world (which would be one world per *entire
run*), multiple worlds per step is the correct default.
**Fix (launch flag only, no code):** worlds/step = `per_device × grad_accum × num_procs / num_generations`
(TRL `RepeatSampler`: `generation_batch_size / num_generations` distinct prompts, `grpo_trainer.py:944`).
Bump `grad_accum 4 → 16` (keep per_device 2, G8) → generation batch 32 → **4 distinct worlds × 8 gens per
step**. grad_accum is sequential so memory is unchanged (75.8 GiB, no OOM). Verified empirically: one
logged step's parquet now has 32 rollouts spanning 4 distinct world_gravity values (6.06/17.64/19.44/21.55).
`rl-mw4-kl005` running (beta 0.005, 250 steps); expect a much smoother reward curve and a lower-variance,
less idiosyncratic gradient. (Run *before* adding the new spin/diameter task, per the plan.)

**Result — fix validated.** Reward step-to-step jitter HALVED (std 0.29→0.14; mean \|Δstep\| 0.23→0.12 —
the ≈1/√4 expected from averaging 4 worlds), grad_norm dropped ~2× (the gradient is far less
idiosyncratic), and it was the most stable run yet (max entropy 0.357, no divergence). It reached
reward 0.76 / aggregation n≈4.7 at step 250 — *below* the 1-world run's 0.88/n7.7, but only because it
had 250 optimizer updates vs 704 (each step is ~4× slower since it generates 4× the rollouts). So
multi-world is the correct setup with a cleaner per-update signal; the only cost is throughput.
**The payoff to cash in:** a 4× lower-variance gradient should tolerate a higher learning rate (we'd been
pinned at lr 2e-6 by *1-world* gradient noise). `rl-mw4-lr4e6` tests lr 4e-6 on the multi-world setup —
if it learns faster per step while staying stable, the throughput penalty disappears. Adopt multi-world
as the standard batching going forward (and for the new task).
